16、软件测试常见问题

1、你为什么选择软件测试岗位？
我选择软件测试岗位是因为我喜欢发现问题和解决问题的过程。测试不仅让我能深入了解软件的每个细节，还能确保用户体验的质量。看到自己发现的bug被修复，心里总是有种成就感。

2、大概说说之前公司的测试流程？
我们公司采用了敏捷开发模式，每个迭代周期会有计划、设计、开发、测试和回归阶段。测试流程从需求分析开始，测试用例设计，测试执行，直到bug跟踪和回归测试。我们使用Jira跟踪bug，所有测试用例都要经过评审，确保覆盖了功能点

 完整测试流程
测试人员的职责：
  1. 获取需求文档，参与需求评审。
  2. 编写测试用例并进行用例评审。
  3. 需求提测，进行三轮测试：单元测试、集成测试和回归测试。
  4. 完成测试后，与产品和UI进行验收，若验收通过则准备发版。
  5. 发版完成后进行线上回测。

测试负责人的职责：
  1. 熟知每个业务需求，制定测试计划。
  2. 进行需求风险评估，进行人员分配。
  3. 参加测试用例评审，对需求进行测试。
  4. 负责进度把控和测试用例的管理分配。

3、测试报告有哪些内容?
测试报告一般包含测试概况、测试环境、测试方法、测试用例执行情况、测试结果分析、发现的Bug、建议和总结。通过这些内容，帮助团队了解当前测试的完成度和质量情况

4、如何保证用例的覆盖度?
我们会通过需求文档和设计文档来设计测试用例，确保每个功能都有对应的用例。同时，使用覆盖率工具来分析用例的覆盖情况，及时补充遗漏的测试用例

5、什么是测试用例,什么是测试脚本?两者的关系
定义：测试用例是测试人员在测试执行之前，根据需求文档编写的详细测试文档。它包含了重要的测试信息，如测试编号、测试目的、测试步骤、实际结果和预期结果。，而测试脚本是自动化测试中执行测试用例的代码。两者关系密切，测试脚本通常是基于测试用例编写的。

6、Bug的级别，按照什么划分？
Bug的级别一般按照严重性和优先级划分，常见的有致命、严重、中等和轻微。致命的bug会导致系统崩溃，而轻微的bug则不会影响主要功能

7、你认为是bug,开发认为不是bug,如何解决?
这种情况可以通过沟通来解决。我会详细说明我的观点，并提供相关的需求或设计文档作为依据，争取达成一致。如果仍然有分歧，可以请项目经理或相关负责人介入

2、get和post的区别
GET 使用URL 或Cookie 传参，而POST将数据放在Body 中。
GET的URL 在长度上会有限制，而POST没有。
POST比GET相对安全，因为在地址栏不可见。
一般POST请求用来获取数据，POST请求用来发送数据。
对于上面的区别，其实第一点POST也可以将数据放在URL里，GET请求其实也没有长度限制，POST请求看起来是隐式的，但是可以通过抓包拿到参数。

对于长度限制主要分两种，一个是浏览器限制（长度限制在2000到8000个字符之间），一个是服务端限制（Nginx是4096，Apache是8190）

5、常用响应状态码含义
1xx：信息性状态码
* 100 Continue：初始请求已接受，客户端可以继续发送请求的其余部分。
* 101 Switching Protocols：服务器正在切换协议，通常用于 WebSocket 连接。
2xx：成功状态码
* 200 OK：请求成功，服务器返回所请求的数据。
* 201 Created：请求成功并创建了新的资源，通常用于 POST 请求。
* 202 Accepted：请求已接受，但尚未处理，通常用于异步处理。
* 204 No Content：请求成功，但没有返回内容，常用于 DELETE 请求。
3xx：重定向状态码
* 301 Moved Permanently：请求的资源已被永久移动到新位置，返回的新 URL 在响应头中提供。
* 302 Found：请求的资源临时移动到新位置，返回的新 URL 在响应头中提供。
* 304 Not Modified：资源未被修改，客户端可以使用缓存的版本。
4xx：客户端错误状态码
* 400 Bad Request：请求无效，服务器无法理解请求的格式。
* 401 Unauthorized：请求未授权，客户端需要提供身份验证信息。
* 403 Forbidden：服务器拒绝请求，客户端没有访问权限。
* 404 Not Found：请求的资源未找到，服务器无法找到指定的 URL。
* 405 Method Not Allowed：请求方法不被允许，客户端使用了不支持的 HTTP 方法。
5xx：服务器错误状态码
* 500 Internal Server Error：服务器内部错误，无法完成请求。
* 501 Not Implemented：服务器不支持请求的功能，无法处理请求。
* 502 Bad Gateway：作为网关或代理的服务器从上游服务器收到无效响应。
* 503 Service Unavailable：服务器当前无法处理请求，通常是由于过载或维护。
* 504 Gateway Timeout：作为网关或代理的服务器未能及时从上游服务器获取响应



6、三次挥手、四次握手
三次握手以及四次挥手原理
三次挥手：
客户端——服务端：客户端发送请求给服务端
服务端——客户端：服务端收到客户端连接
客户端——服务端：客户端发送消息给服务端
四次挥手：
主动方——被动方：主动方告诉被动方断开连接
被动方——主动方：被动方收到断开连接请求
被动方——主动方：被动方准备断开连接发送给主动方
主动方——被动方：主动方断开连接

1、性能测试里面的参数指标?
 响应时间 (Response Time)
* 定义：指从发送请求到接收到响应所需的时间。
* 关注点：包括平均响应时间、最大响应时间和最小响应时间。理想情况下，响应时间应保持在可接受的范围内，以确保良好的用户体验。
吞吐量 (Throughput)
* 定义：单位时间内成功处理的请求数量，通常以请求/秒 (requests per second) 或事务/秒 (transactions per second) 表示。
* 关注点：吞吐量越高，表示系统在高负载下的处理能力越强。需要监测在不同负载下的吞吐量变化。
错误率 (Error Rate)
* 定义：在测试过程中发生的错误请求数量与总请求数量的比率，通常以百分比表示。
* 关注点：高错误率可能表明系统存在问题，需要进一步分析错误的原因。
并发用户数 (Concurrent Users)
* 定义：同时发起请求的用户数量。
* 关注点：通过增加并发用户数来测试系统的承载能力，观察系统在高并发情况下的表现。
CPU 使用率 (CPU Utilization)
* 定义：服务器 CPU 的使用情况，通常以百分比表示。
* 关注点：监测在负载测试期间 CPU 使用率的变化，确保系统不会因 CPU 资源不足而导致性能下降。
内存使用率 (Memory Utilization)
* 定义：服务器内存的使用情况，通常以百分比表示。
* 关注点：监测内存使用情况，确保在高负载下不会出现内存泄漏或不足的问题。
磁盘 I/O (Disk I/O)
* 定义：磁盘读写操作的速率，通常以字节/秒或操作/秒表示。
* 关注点：监测磁盘 I/O 性能，确保在高负载下不会成为系统瓶颈。
网络带宽 (Network Bandwidth)
* 定义：网络传输的速率，通常以字节/秒表示。
* 关注点：监测网络带宽的使用情况，确保在高并发请求下网络不会成为瓶颈。
事务响应时间 (Transaction Response Time)
* 定义：特定事务（如用户登录、订单提交等）的响应时间。
* 关注点：监测关键业务流程的响应时间，确保用户体验良好。
 连接数 (Number of Connections)
* 定义：同时打开的连接数量。
* 关注点：监测在高负载下的连接数，确保服务器能够处理足够的并发连接。
服务器状态 (Server Status)
* 定义：服务器的健康状态，包括 CPU、内存、磁盘和网络的实时监控。
* 关注点：确保服务器在测试过程中保持稳定，及时发现潜在问题

3、负载测试和压力测试的区别?负载/压力测试应该关注什么?
* 目的：
    * 负载测试：主要目的是评估系统在正常和高负载条件下的性能表现。它通过模拟多个用户同时访问系统，来确定系统在预期负载下的响应时间、吞吐量和资源使用情况。
    * 压力测试：主要目的是评估系统在超出正常负载条件下的表现，目的是找出系统的最大承载能力和瓶颈。它通过逐渐增加负载，直到系统崩溃或出现故障，以确定系统的极限。
* 测试场景：
    * 负载测试：通常模拟正常的用户行为，关注系统在高并发情况下的稳定性和响应时间。例如，模拟1000个用户同时登录并进行操作。
    * 压力测试：通常模拟极端情况，关注系统在高于正常负载的情况下的表现。例如，模拟5000个用户同时登录，甚至更高的并发量。
* 结果分析：
    * 负载测试：关注系统在正常负载下的性能指标，如响应时间、错误率、资源利用率等，确保系统能够满足用户需求。
    * 压力测试：关注系统的稳定性和恢复能力，分析系统在崩溃后的恢复时间和数据完整性。


* 响应时间：无论是负载测试还是压力测试，响应时间都是一个关键指标。需要监测在不同负载下的平均响应时间和最大响应时间。
* 吞吐量：指单位时间内处理的请求数量。负载测试中需要确保在正常负载下吞吐量达到预期，而压力测试中则需要观察在极限负载下的吞吐量变化。
* 错误率：监测在负载或压力测试过程中出现的错误数量，确保系统在高负载下仍能保持低错误率。
* 资源利用率：监测CPU、内存、磁盘和网络等资源的使用情况，确保系统在高负载下不会出现资源瓶颈。
* 系统稳定性：在压力测试中，观察系统在超负载情况下的稳定性，记录崩溃、挂起或其他异常情况。
* 恢复能力：测试系统在压力测试后能否快速恢复到正常状态，确保数据的完整性和可用性。
* 用户体验：在负载测试中，关注用户的实际体验，包括页面加载时间、操作流畅性等

5、压力、并发、负载这三个有什么区别?
压力测试关注系统的极限承载能力；并发测试关注多个用户同时操作的表现；负载测试则是测试系统在正常负载下的性能。

6、如果下一个接口对上一个接口有依赖的时候该怎么做?
在测试时，可以先执行第一个接口，确保其返回结果正确后，再执行依赖的接口，确保整个流程的连贯性


实际测试类
1、给你一个纸杯,如何测试?
* 耐热性测试：将纸杯装满热水（例如90°C），观察纸杯在一定时间内是否变形或漏水。
* 耐压性测试：用手施加压力，检查纸杯在不同压力下的变形情况，确保其在正常使用中不会破裂。
* 密封性测试：装满水后，盖上杯盖，观察是否有漏水现象，确保在运输或使用过程中不会漏水。
* 材料测试：检查纸杯的材料是否符合食品安全标准，确保无有害物质。
* 耐久性测试：反复使用纸杯，观察其在多次使用后的表现，确保其在短时间内不会损坏。


2、电梯的测试点(小米面试题现场发挥)
* 载重测试：在电梯内加载不同重量的物品，测试电梯在最大载重下的运行情况，确保其能够安全承载。
* 开关门时间：测试电梯门的开关时间，确保其在合理范围内，避免用户被夹伤。
* 上下运行平稳性：观察电梯在上升和下降过程中的平稳性，确保没有剧烈晃动或停顿。
* 紧急制动功能：模拟紧急情况，测试电梯的紧急制动功能是否正常，确保在故障时能够及时停止。
* 安全警报：测试电梯内的安全警报系统，确保在发生故障时能够及时发出警报。


3、给你一支笔，你如何测试?(小米笔试题)
* 书写流畅性：在不同类型的纸张上进行书写，观察墨水是否流畅，是否有断墨现象。
* 墨水干燥时间：测试书写后墨水干燥的时间，确保在正常使用中不会出现污迹。
* 耐用性测试：在纸上进行长时间书写，观察笔尖是否磨损，墨水是否能持续输出。
* 不同纸张测试：在光滑、粗糙、厚薄不同的纸张上进行书写，确保笔在各种条件下都能正常使用。
* 防漏测试：将笔倒置，观察是否有墨水漏出，确保在携带过程中不会弄脏物品。


4、微信视频的测试点
* 视频清晰度：测试不同网络条件下的视频清晰度，确保在高、低带宽下都能保持良好的画质。
* 延迟测试：测量视频播放的延迟时间，确保用户在观看时不会感到卡顿。
* 音视频同步：观察音频与视频的同步情况，确保二者在播放时没有明显的延迟。
* 网络波动测试：模拟网络波动，观察视频播放的稳定性，确保在网络不稳定时能够自动调整清晰度。
* 多平台兼容性：在不同设备（手机、平板、电脑）上测试视频播放，确保兼容性良好。


5、给你一个登录框你怎么测试?
* 输入验证：测试不同类型的输入（如空值、特殊字符、过长字符）是否能被正确处理。
* 错误提示：故意输入错误的用户名或密码，观察系统是否能给出明确的错误提示。
* 密码强度：测试不同强度的密码（如简单密码、复杂密码）是否能被系统接受，确保密码强度要求合理。
* 不同用户角色的登录权限：测试不同角色（管理员、普通用户）登录后的权限，确保权限控制正常。
* 会话管理：测试登录后会话的有效性，确保在一定时间后自动登出，保护用户信息安全


6、优惠卷怎么测试?
* 有效性测试：测试不同类型的优惠券（如满减、折扣）在有效期内是否能正常使用。
* 使用条件：测试优惠券的使用条件（如最低消费、特定商品）是否能被正确识别。
* 叠加使用情况：测试是否可以叠加使用多个优惠券，确保系统能正确计算最终价格。
* 过期处理：测试过期优惠券是否能被系统拒绝，确保用户无法使用过期的优惠。
* 优惠券的生成和分发：测试优惠券的生成逻辑和分发渠道，确保用户能顺利获取。


7、说一下添加到购物车的测试点
* 数量限制：测试添加商品时的数量限制，确保用户不能添加超过库存的商品。
* 库存检查：测试在添加商品时，系统是否能实时检查库存，避免超卖。
* 价格计算：测试购物车中商品的总价计算，确保计算准确，包括优惠和运费。
* 购物车的持久性：测试在不同设备和浏览器中，购物车内容是否能保持一致。
* 删除和修改功能：测试用户能否顺利删除或修改购物车中的商品，确保操作简单易用


8、支付功能怎么测试(银行项目)
* 支付流程完整性：测试从选择商品到支付完成的整个流程，确保每一步都能正常进行。
* 支付方式的多样性：测试不同支付方式（如信用卡、支付宝、微信支付）是否都能正常使用。
* 支付成功后的反馈：测试支付成功后的反馈信息，确保用户能清晰地看到支付结果。
* 异常处理：测试在支付过程中出现异常（如网络中断、支付失败）时，系统的处理能力。
* 安全性测试：测试支付过程中的数据加密和安全性，确保用户信息不被泄露。


9、直播功能怎么测试?
* 流畅性测试：测试直播过程中视频的流畅性，确保没有卡顿或延迟。
* 延迟测试：测量直播的延迟时间，确保用户观看时的实时性。
* 观众互动功能：测试评论、点赞等互动功能是否正常，确保用户能参与互动。
* 多平台兼容性：在不同设备和平台上测试直播功能，确保兼容性良好。
* 网络波动测试：模拟网络波动，观察直播的稳定性，确保在网络不稳定时能够自动调整清晰度


10、购物车功能怎么测试?
* 添加商品功能：测试用户能否顺利将商品添加到购物车，确保操作简单。
* 删除商品功能：测试用户能否顺利删除购物车中的商品，确保操作无误。
* 修改商品数量：测试用户能否修改购物车中商品的数量，确保数量计算准确。
* 结算流程完整性：测试从购物车到结算的整个流程，确保每一步都能正常进行。
* 优惠券应用：测试在结算时应用优惠券的功能，确保优惠计算准确。
